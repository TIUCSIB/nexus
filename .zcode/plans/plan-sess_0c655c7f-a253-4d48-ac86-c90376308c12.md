## 6 个优化项实施计划

### 1. 日志轮转
**方案**: 文件日志 + 按大小轮转
- 新建 `agent/internal/logger/rotator.go`
- `RotatingFileWriter`: 写入文件，超 10MB 自动切换 `agent.log.1`、`agent.log.2`（最多保留 3 份）
- `agent/config.yml.example` 添加 `log_file`、`log_max_size` 配置项
- `agent/config/config.go` 添加对应的字段
- `agent/cmd/agent/main.go`: 如果配置了 `log_file`，`log.SetOutput(logger.NewRotatingFileWriter(path, maxSize))`
- 改动用 3 个文件，约 60 行

### 2. 自动更新
**方案**: WebSocket "update" 命令 → 下载新二进制 → 覆盖 → 重启
- 传统模式 main.go 的 WS handler 添加 "update" 命令
- 从 Data 中提取 `download_url`，下载到临时文件，校验后覆盖自身，然后重启
- 面板端不需要改动（已支持 WS 命令下发）
- 改动用 1 个文件，约 50 行

### 3. 健康检查端点
**方案**: 在 Agent 的 StatsPort 端口上添加 `/health` 路由
- 实际上 sing-box 的 Clash API 端口已经有了（StatsURL）
- 直接在 main.go 和 orchestrator.go 的 stats ticker 位置添加一个简单的 health check HTTP server 不太合适
- 更好的方法：使用现有的 stats port，在启动时加一个独立的 goroutine 跑一个 tiny HTTP server，只响应 `/health`
- 或者直接用 sing-box 的 stats API + 我们自己暴露一个额外端口
- 最简单方案：在 `Config` 中加入 `health_port` 字段，启动一个简单的 HTTP 服务（只有一个 `/health` 端点）
- 改动用 2 个文件，约 40 行

### 4. 延迟测试
**方案**: 心跳时附带节点延迟（TCP ping）
- 在 `httpclient/client.go` 的 `Heartbeat()` 方法中，计算从 agent 到 `node address:port` 的 TCP 连接时间
- 或者作为一个独立函数，在心跳时调用
- 但问题在于：agent 是运行在节点机器上的，节点地址通常是 127.0.0.1 或内网地址，延迟没有意义
- 实际上节点的延迟应该由**用户客户端**来测试，而不是 agent 自测
- **结论：延迟测试对 agent 无意义，跳过**

### 5. WS 重连后推送配置（机器模式）
**方案**: `OnReconnect` 回调中，除了 `rediscover()`，再为每个已运行节点拉取最新配置并 hot-reload
- 修改 `agent/internal/machine/orchestrator.go` 的 `OnReconnect`
- 遍历 `o.nodes`，对每个节点调用 `pullNodeConfig()` + `hotReloadOrRestartNode()`
- 改动用 1 个文件，约 25 行

### 6. 流量缓冲上限优化
**方案**: 增大上限 + 改为丢弃最旧数据而非全部清空
- 将 `maxPendingUsers` 从 1000 提升到 5000
- 当溢出时，改为删除最早的一半条目，而不是全部替换
- 同时日志记录更加清晰（记录实际丢弃数）
- 改动用 1 个文件（main.go 和 orchestrator.go 各一处），约 15 行

### 实施顺序
1. 日志轮转
2. 自动更新
3. 健康检查
4. WS 重连推送
5. 流量缓冲

延迟测试跳过，原因：agent 自测本机节点延迟没有意义，应该由用户客户端测。
